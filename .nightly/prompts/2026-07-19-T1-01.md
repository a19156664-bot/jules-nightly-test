# T1-01: state_manager.py を config.py 参照にリファクタ（定数の二重管理解消）

## 背景

`commander/state_manager.py` は状態ファイルのパスと予算の上限値
（日次上限・5時間ウィンドウ上限・ウィンドウ時間）を内部にハードコードしています。
一方、`commander/config.py`（前夜に作成済み）は同じ値を定数として定義しています。
現在は両者の値が偶然一致していますが、二重管理のままでは将来どちらか一方だけを
変更したときに不整合が発生します。state_manager.py 側を config.py 参照に統一します。

## 変更するファイル

- `commander/state_manager.py`（修正）
- `tests/test_state_manager.py`（必要な範囲で修正・追加）

## 変更禁止ファイル

- `commander/config.py`（変更しない。参照のみ）
- `commander/commander.ps1` / `commander/create_alert_issue.py`
- `.nightly/` 配下すべて / `.github/` 配下すべて / `AGENTS.md`
- 既存テストの削除・スキップ・期待値の緩和は禁止（テストの「追加」は可）

## 仕様

1. `commander/config.py` から以下を import して使用すること:
   - `STATE_FILE`（状態ファイルのパス）
   - `MAX_LLM_CALLS_PER_DAY`（日次上限）
   - `MAX_LLM_CALLS_PER_WINDOW`（ウィンドウ上限）
   - `WINDOW_HOURS`（ウィンドウ時間）
2. state_manager.py 内にハードコードされていた同等の値・パスをすべて削除し、
   config の定数参照に置き換えること。
3. 数値のフォールバック（config が読めない場合に内蔵デフォルトへ戻る等）は
   実装しないこと。config が唯一の定義源である状態にする。
4. 外部から見た振る舞い（CLI の引数・出力形式・exit code）は一切変えないこと:
   - `--can-call-llm` / `--should-stop` / `--check-turn-due` は "True" / "False" を stdout に出力
   - `--record-wakeup` / `--record-llm-call` は "OK" を出力
   - `--get KEY` はドット区切りのネストキーに対応
   - exit code は従来どおり
5. import は相対 import ではなくパッケージとして解決できる形にすること
   （既存のテストと CLI 実行 `python commander/state_manager.py ...` の両方が
   動く形を維持する。必要なら sys.path 調整は state_manager.py 側で行ってよい）。

## テスト要件

- 既存の `tests/test_state_manager.py` の全テストがそのまま通ること。
- 追加テスト: state_manager が参照する上限値が config.py の定数と一致することを
  検証するテストを最低1つ追加すること
  （例: `state_manager` モジュール経由で観測される上限が
  `config.MAX_LLM_CALLS_PER_DAY` と等しい、など。値のベタ書き比較は避ける）。

## 完了条件

1. `python -m pytest tests/test_state_manager.py` が全件パス
2. `python commander/state_manager.py --can-call-llm` が従来どおり "True"/"False" を出力
3. state_manager.py 内に予算上限値・ウィンドウ時間・状態ファイルパスの
   ハードコードが残っていない（config.py 参照のみ）
4. commander/config.py に変更がない
