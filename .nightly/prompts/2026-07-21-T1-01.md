# T1-01: parse_output.py に proposal ファイル書き出し機能を追加

## 背景

司令塔（LLM）は `commander/proposals/` にファイルを書き出したいが、
テキスト応答しか返せない。現在の `commander/parse_output.py` は
ACTION/RESULT 等のフィールドだけをパースしている。

本タスクでは、LLM 出力に含まれる **PROPOSAL_FILE ブロック** を検出し、
指定パスにファイルとして書き出す機能を `parse_output.py` に追加する。

## 完了条件

1. `commander/parse_output.py` が LLM 出力中の以下の形式を検出し、
   指定パスにファイルを書き出すこと:

   ```
   PROPOSAL_FILE: commander/proposals/2026-07-22/tasks.yml
   <<<CONTENT
   night: "2026-07-22"
   turn1:
     - id: "T1-01"
       title: "..."
   >>>END_PROPOSAL
   ```

   形式の仕様:
   - `PROPOSAL_FILE: <relative-path>` 行でブロック開始。パスは相対パス
   - 次の行の `<<<CONTENT` でファイル内容の開始
   - `>>>END_PROPOSAL` 行でブロック終了
   - 1つの出力に複数ブロックが含まれる場合がある（通常3つ: tasks.yml, T1-01.md, T2-01.md）
   - ブロック間や前後に通常のテキスト（ACTION/RESULT 等）が混在してよい

2. **安全制約**: 書き出し先パスが `commander/proposals/` 配下であることを検証すること。
   それ以外のパス（特に `.nightly/`, `.github/`）への書き込みは拒否し、
   標準エラーに警告を出力して**そのブロックだけスキップ**すること（他のブロックは処理を続行）。
   パストラバーサル（`..` を含むパス）も拒否すること。

3. 書き出し先ディレクトリが存在しない場合は自動作成すること（`mkdir -p` 相当）。

4. proposal ブロックが0件の場合（通常の ACTION/RESULT のみの出力）でも
   既存動作に影響を与えないこと（後方互換性）。

5. 書き出し成功時は stdout に `PROPOSAL: wrote <path>` を出力すること。

6. テストを `tests/test_parse_output.py`（新規作成）に追加すること:
   - proposal ブロック1件の正常書き出し
   - proposal ブロック3件（tasks.yml + T1-01.md + T2-01.md）の正常書き出し
   - 不正パス（`.nightly/` 配下）のブロックが拒否されること
   - パストラバーサル（`..` を含むパス）のブロックが拒否されること
   - proposal ブロックなし（従来形式のみ）で既存動作が壊れないこと
   - ACTION/RESULT と proposal ブロックが混在する出力の正常処理

## 変更可能ファイル

- `commander/parse_output.py`
- `tests/test_parse_output.py`（新規作成）

## 変更禁止ファイル

- `.nightly/` 配下すべて
- `.github/` 配下すべて
- `AGENTS.md`
- `commander/state_manager.py`
- `commander/config.py`
- `commander/build_prompt.py`

## 技術的注意事項

- `parse_output.py` は `python -m commander.parse_output <log-file>` で実行される
- 既存の `parse_output()` 関数と `update_state()` 関数は変更しないこと。
  新しい関数（例: `extract_proposals()`, `write_proposals()`）を追加し、
  `main()` 関数の中で既存処理の後に呼び出す構成にすること
- パスの解決はリポジトリルート（`Path(__file__).resolve().parent.parent`）を基準にする
- ファイル書き込みは UTF-8 エンコーディングを使用すること
