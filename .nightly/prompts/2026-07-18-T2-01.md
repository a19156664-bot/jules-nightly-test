# T2-01: commander.ps1 雛形 + LOOP-ALERT Issue 作成スクリプトの新規作成

## 背景
「司令塔」はローカル Windows PC のタスクスケジューラから30分ごとに起動される
PowerShell スクリプトです。LLM の呼び出しコストを節約するため「2段ゲート方式」を
採用します: Stage 0（LLM を呼ばない事前判定）で仕事がなければ即終了し、
仕事がある場合のみ Stage 1（LLM 呼び出し）に進みます。

本タスクでは (A) commander.ps1 の雛形と (B) 異常通知用の Issue 作成スクリプトを
新規作成します。turn1 でマージ済みの `commander/state_manager.py` を利用します。

## 作成するファイル（3つ）
1. `commander/commander.ps1`（新規）
2. `commander/create_alert_issue.py`（新規）
3. `tests/test_create_alert_issue.py`（新規）

## (A) `commander/commander.ps1` の仕様

PowerShell 5.1 互換で書くこと（PowerShell 7 専用構文は使わない）。

### 構成

```powershell
# ===== 設定 =====
$ErrorActionPreference = "Stop"
$env:PYTHONIOENCODING = "utf-8"
$RepoRoot   = $PSScriptRoot | Split-Path -Parent   # commander の親 = リポジトリルート
$Model      = "claude-opus-4-8"                     # 司令塔モデル（変数化）
$MaxTimeout = 300                                   # LLM 呼び出しタイムアウト秒

# ===== Stage 0: LLM を呼ばない事前判定 =====
# 1. 日次リセット + wakeup 記録
# 2. 恒久停止チェック: state_manager.py --should-stop が True なら即終了
# 3. 仕事の有無チェック（gh CLI）:
#    $openPRs   = gh pr list --state open --json number --jq 'length'
#    $alertOpen = gh issue list --label loop-alert --state open --json number --jq 'length'
#    $turnDue   = python commander/state_manager.py --check-turn-due
# 4. 仕事ゼロなら record-wakeup "no-work" して exit 0

# ===== 予算チェック（ソフトスキップ）=====
# state_manager.py --can-call-llm が False なら
# record-wakeup "budget-deferred" して exit 0（停止ではなく持ち越し）

# ===== Stage 1: LLM 呼び出し =====
# 1. 憲法(.nightly/COMMANDER.md)と state.yml を読み込んでプロンプトを組み立てる
#    ※ COMMANDER.md が存在しない場合は
#      「[SKIP] COMMANDER.md not found」と出力して exit 0（雛形段階の安全弁）
# 2. claude -p にプロンプトを渡す:
#    $result = $prompt | claude -p --model $Model
# 3. 成功したら state_manager.py --record-llm-call
# 4. 結果を commander/logs/commander-YYYYMMDD-HHmmss.log に UTF-8 で保存

# ===== エラーハンドリング =====
# try/catch で全体を包み、catch では:
#   - エラー内容をログファイルに保存
#   - python commander/create_alert_issue.py --alert-type runtime-error
#       --summary "commander.ps1 crashed" --detail "<エラーメッセージ>"
#   - exit 1
```

### 注意
- リポジトリルートは空白を含むパスに置かれる可能性がある。パスは常にダブルクォートで囲むこと
- `claude` CLI と `gh` CLI はインストール済みの前提でよい
- このスクリプトは**雛形**であり、Stage 1 のプロンプト組み立ての詳細は後日調整される。
  構造とゲート処理が正しく動くことを優先すること
- ログディレクトリ `commander/logs/` が存在しない場合は自動で作成すること

## (B) `commander/create_alert_issue.py` の仕様

GitHub Issue を作成して人間に異常を通知するスクリプト。

### CLI

```
python commander/create_alert_issue.py \
  --alert-type <type> --summary "<1行要約>" [--detail "<詳細>"]
```

- `--alert-type` の許容値: `budget-exceeded` / `error-limit` /
  `protected-path-violation` / `runtime-error` / `manual-stop`
  （それ以外はエラー終了 exit 2）
- `--summary`: 必須。1行要約
- `--detail`: 任意。省略時は空

### 動作
1. Issue タイトル: `[LOOP-ALERT] {alert_type}: {summary}`
2. Issue 本文:
   - `## Detail` セクションに --detail の内容
   - `## State Snapshot` セクションに `commander/state.yml` の現在値をコードブロックで添付
     （state.yml が読めない場合は "state.yml unavailable" と記載して続行）
   - 発生時刻 (JST, ISO8601)
3. `subprocess` で `gh issue create --title ... --body ... --label loop-alert` を実行
4. ラベル `loop-alert` が存在しない場合も考慮:
   `gh issue create` が失敗したら `--label` なしで1回だけリトライする
5. 成功時: 作成された Issue URL を標準出力に1行出力、exit 0
6. 失敗時: エラーを標準エラーに出力、exit 1

### 実装上の制約
- 依存は標準ライブラリ + PyYAML のみ
- 同一 alert-type の open Issue が既に存在する場合は新規作成せず、
  既存 Issue にコメントを追加する（`gh issue comment`）。通知の重複を防ぐため

## テスト
`tests/test_create_alert_issue.py` を作成し、pytest で以下を検証すること。
gh コマンドの実行は `unittest.mock.patch` で subprocess をモックすること（実 API は叩かない）:
1. 不正な --alert-type で exit 2
2. タイトルと本文が正しく組み立てられる（モックに渡された引数を検証）
3. gh 失敗時にラベルなしでリトライされる

commander.ps1 は PowerShell のため自動テスト対象外でよい。
