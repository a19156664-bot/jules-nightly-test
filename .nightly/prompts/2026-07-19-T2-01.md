# T2-01: pr_checker.py の新規作成（レビュー機械チェック3項目の自動化）

## 背景

司令塔（Claude Code）は PR レビュー時に6項目をチェックしますが、そのうち3項目は
機械的に判定可能です。これを Python スクリプトとして切り出すことで、司令塔の
LLM レビュー前の事前フィルタとして使い、LLM 予算の節約とレビュー品質の安定化を
図ります。

## 作成するファイル

- `commander/pr_checker.py`（新規作成）
- `tests/test_pr_checker.py`（新規作成）

## 変更禁止ファイル

- `commander/` 配下の既存ファイル（config.py / state_manager.py /
  commander.ps1 / create_alert_issue.py）
- `.nightly/` 配下すべて / `.github/` 配下すべて / `AGENTS.md`
- 既存テストの削除・スキップ・期待値の緩和は禁止

## 仕様

### 判定する3項目

変更ファイルのパス一覧（と、必要に応じてタスク定義）を入力として、
以下を判定する:

1. **protected_paths 違反**: 変更ファイルに `.nightly/`, `.github/`, `AGENTS.md`
   配下のものが含まれていないか
2. **スコープ逸脱**: 変更ファイルがすべて、タスクで宣言された許可パス
   （paths のリスト）の配下にあるか
3. **.egg-info 混入**: 変更ファイルに `.egg-info` を含むパスがないか

### 実装要件

- 純粋関数として実装すること。**gh CLI や GitHub API を内部で呼ばない**。
  入力（変更ファイルのリスト、許可パスのリスト）は引数で受け取る。
  ネットワークやサブプロセスに依存しないことでテスト容易性を保つ。
- `commander/config.py` の定数を参照できる場合は参照する
  （protected の既定リストなど。config に該当定数がなければ pr_checker.py 内に
  定義してよいが、その場合はモジュール先頭に定数として明示すること）。
- 判定結果は構造化して返すこと（例: dataclass または dict で
  `ok: bool` / `violations: list[str]`（人間可読な理由文字列のリスト））。
- CLI としても使えること:
  `python commander/pr_checker.py --files a.py b.py --allowed webui/ tests/`
  のような形式で、判定結果を表示し、全項目パスなら exit 0、違反ありなら exit 1。
- Windows パス（バックスラッシュ）と POSIX パスの両方の入力を正しく扱うこと。

## テスト要件

`tests/test_pr_checker.py` に最低限以下のケースを含めること:

1. protected 違反あり（`.nightly/tasks.yml` の変更）→ 違反として検出
2. protected 違反あり（`.github/workflows/x.yml`）→ 違反として検出
3. スコープ逸脱（許可 `webui/` のみで `commander/x.py` を変更）→ 違反として検出
4. .egg-info 混入 → 違反として検出
5. すべて正常（許可パス内のみ・protected なし・egg-info なし）→ ok
6. 複数違反の同時検出（violations に複数件が入る）

## 完了条件

1. `python -m pytest tests/test_pr_checker.py` が全件パス
2. CLI 実行で exit code が仕様どおり（正常 0 / 違反 1）
3. 外部コマンド・ネットワークへの依存がない
4. 既存ファイルへの変更がない
