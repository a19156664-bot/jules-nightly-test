# T2-01: validate_proposal に 1タスクあたりの paths 数上限(3)検査を追加

## 背景

タスク設計基準（COMMANDER.md）では「**1タスクの変更ファイルは3つ以下を推奨**」と
定めている。変更ファイルが増えるほどレビュー範囲が広がり、Jules の手戻りや
スコープ逸脱のリスクが上がるため、本システムでは3を上限として扱う方針とする。

しかし `validate_proposal.py` は `paths` が protected_path を含むか等は検査するが、
**paths の件数**は検査していない。本タスクでは、1タスクの `paths` が3を
超える場合にエラーとして検出する処理を追加する。

※ 本タスクは T1-01 が integration ブランチへマージされた後に着手される
（turn2 は turn1 完了後に実行されるため、T1-01 の変更は取り込み済みの前提でよい）。

## 完了条件

1. `validate_proposal(proposal_dir_str)` に、各タスクの `paths` が
   **リストであり、かつ要素数が 3 を超える**場合にエラーを追加する検査を
   加えること。エラーはタスクごとに1件とする
   （例: `Task T1-01 in turn1 has 4 paths, exceeding the maximum of 3`）。

2. `paths` が欠落している、またはリストでない場合は、既存の該当エラー
   （missing required fields / 非リスト時の扱い）に任せ、本件数検査では
   追加のエラーを出さないこと（`paths` が list のときのみ件数を判定する）。

3. 既存の検証項目・出力形式・exit code は変更しないこと。新チェックの失格も
   `VALIDATION: FAIL` + `- <理由>` の複数行形式で報告されること。

4. 既存テストを壊さないこと。`tests/test_validate_proposal.py` に以下を追加:
   - paths を4件持つタスクを含む proposal が FAIL し、理由に上限超過が
     示されること。
   - paths を3件持つタスク（境界値）は本検査では FAIL しないこと。
   - 既存の正常 proposal（paths 1〜2件）が引き続き PASS すること。

## 技術的注意事項

- 実行は `python -m commander.validate_proposal <dir>` 形式。
- 上限は 3（`paths` の要素数が 4 以上で FAIL）。マジックナンバーは
  モジュール先頭の定数（例: `MAX_PATHS_PER_TASK = 3`）として定義すると可読性が高い。
- 既存の paths ループ（protected 判定）と同じ `isinstance(paths, list)` の
  ガード下で件数を判定できる。protected 判定ループの前後どちらでもよいが、
  件数エラーとパス個別エラーが両方出ても構わない。
- 既存テストの paths は最大2件なので、本検査で既存テストが FAIL しない想定。
  意図せず FAIL する場合は実装側の副作用を疑うこと
  （既存テストの削除・スキップ・期待値変更は禁止）。
- 既存モジュールは import・変更しない。本スクリプトは自己完結のまま保つこと。
