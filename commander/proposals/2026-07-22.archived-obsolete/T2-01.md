# T2-01: commander.ps1 Row 7 で proposal 書き出し後に validate_proposal を実行

## 背景

判断表 Row 7 実行後、`parse_output.py` が LLM 出力から PROPOSAL_FILE ブロックを
検出して `commander/proposals/YYYY-MM-DD/` にファイルを書き出す（PR #24）。

T1-01 で新設される `commander/validate_proposal.py` は、書き出された
proposal ディレクトリを検証するツールである。本タスクでは、司令塔の起動スクリプト
`commander.ps1` において、**proposal 書き出し直後に自動で検証を実行**し、
結果をログへ残すようにする。これにより、人間が承認前に proposal の
妥当性を一目で確認できるようになる。

※ 本タスクは T1-01（validate_proposal.py）が integration ブランチへ
マージされた後に着手される（turn2 は turn1 完了後に実行されるため、
`commander/validate_proposal.py` は既に存在する前提でよい）。

## 完了条件

1. `commander.ps1` の中で、`parse_output.py` の実行が成功した後に、
   当該起動で proposal が書き出されたかを判定し、書き出された場合のみ
   `validate_proposal.py` を実行すること。

2. proposal が書き出されたかの判定は、`parse_output.py` の stdout に含まれる
   `PROPOSAL: wrote <path>` 行の有無で行うこと（PR #24 で追加された出力）。
   1件以上あれば、その書き出し先ディレクトリ（`commander/proposals/YYYY-MM-DD`）を
   特定して検証対象とすること。

3. 検証は以下の形式で実行すること:
   ```powershell
   & $Python -m commander.validate_proposal $proposalDir
   ```
   `$Python` はスクリプト冒頭で定義済みの変数を用いること。

4. 検証結果（stdout の `VALIDATION: PASS` / `VALIDATION: FAIL ...` と exit code）を
   commander.ps1 のログへ出力すること。

5. **検証は log-only（非ブロッキング）**とすること。VALIDATION: FAIL であっても
   commander.ps1 自体は異常終了させず、state.yml の stop_reason や error_count を
   変更しないこと（proposal の採否は人間が判断するため）。ログに
   警告として記録するに留めること。

6. proposal が1件も書き出されなかった通常の起動（ACTION/RESULT のみ）では、
   validate_proposal を呼ばず、既存動作に一切影響を与えないこと。

7. 検証手順を PR の説明に記載すること（自動テストは不要）:
   - `commander/proposals/` にダミー proposal を書き出す LLM 出力を用意して
     commander.ps1 を実行し、ログに `VALIDATION:` 行が出力されることを確認する手順。

## 技術的注意事項

- commander.ps1 は PowerShell 5.1 で実行される。
- 日本語出力の文字化け対策として OutputEncoding は UTF-8 設定済み（コミット 53e1a77）。
  新規の Write-Host / ログ出力もこれに従うこと。
- `parse_output.py` の stdout はログファイルへ記録されている想定。既に stdout を
  変数へ受けている箇所があればそれを再利用し、無ければ最小限の追加に留めること。
- 変更は proposal 書き出し（parse_output 実行）直後のブロックに限定すること。
  Stage 0 / Stage 1 の既存ロジックや LLM 呼び出し部分は変更しないこと。
- state_manager.py は呼び出しても値変更（--set 等）を行わないこと。読み取りのみ。
