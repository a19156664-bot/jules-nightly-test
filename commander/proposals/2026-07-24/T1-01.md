# T1-01: validate_proposal に task id の turn 整合と一意性検査を追加

## 背景

`commander/validate_proposal.py` は各タスクの必須フィールド（id, title, risk,
paths, prompt_file）の存在は検証するが、**task id の値そのものの妥当性**は
検証していない。

タスク設計基準（COMMANDER.md）では、タスクIDは `T1-01`（turn1）, `T2-01`（turn2）
のように、**turn 番号と id の接頭辞が対応**することになっている。また、同一
proposal 内で **id が重複してはならない**。id のコピペミス（turn2 に `T1-01` を
書く、両ターンで同じ id を使う等）は現実に起こり得るが、現状は素通りする。

本タスクでは、この2点（turn との接頭辞整合・proposal 全体での一意性）を
`validate_proposal.py` に機械的検査として追加する。

## 完了条件

1. `validate_proposal(proposal_dir_str)` に、次の2つの検証を追加すること:

   (a) **turn 接頭辞の整合**: `turn1` の各タスク id は `T1-` で始まり、
       `turn2` の各タスク id は `T2-` で始まること。違反時は errors に
       1件ずつ追加する
       （例: `Task T2-05 in turn1 has id that does not match turn prefix 'T1-'`）。

   (b) **id の一意性**: proposal 全体（turn1 と turn2 を通算）で id が
       重複しないこと。重複した id ごとに errors に1件追加する
       （例: `Duplicate task id across proposal: T1-01`）。

2. どちらの検査も、id が **文字列として存在する場合のみ**適用すること。
   id が欠落しているタスクは既存の「missing required fields」エラーに任せ、
   本検査では追加のエラーを出さない（欠落 id を一意性判定の対象にしない）。

3. 既存の検証項目・出力形式（`VALIDATION: PASS` / `VALIDATION: FAIL` +
   `- <理由>` の複数行出力、exit code 0/1）は一切変更しないこと。
   新チェックの失格も同じ形式で報告されること。

4. 既存テストを壊さないこと。`tests/test_validate_proposal.py` に以下を追加:
   - turn2 のタスク id を `T1-99` にした proposal が FAIL し、理由に
     turn 接頭辞不整合が示されること。
   - turn1 と turn2 の id を同一（例: 両方 `T1-01`）にした proposal が FAIL し、
     理由に重複が示されること。
   - id が正しく turn と対応し重複もない既存の正常 proposal が
     引き続き PASS すること（`valid_tasks_data` はそのまま利用可）。

## 技術的注意事項

- 実行は `python -m commander.validate_proposal <dir>` 形式。
- 接頭辞は turn 名から機械的に導出すること（`turn1` → `T1-`, `turn2` → `T2-`）。
  ハードコードの分岐でも `f"T{turn[-1]}-"` のような導出でもよいが、可読性を優先する。
- 一意性判定では、turn1 と turn2 のループをまたいで id を集計する必要がある。
  既存の turn ごとのループ構造を大きく変えず、収集用の集合／リストを
  ループ外に用意して蓄積する形が無難。
- id が欠落（None）のタスクは接頭辞・一意性いずれの対象にもしないこと。
- 既存の `valid_tasks_data` フィクスチャは turn1=`T1-01`, turn2=`T2-01` で
  整合・一意なので、既存テストは追加要件でもそのまま PASS する想定。
  意図せず既存テストが FAIL する場合は実装側の副作用を疑うこと
  （既存テストの削除・スキップ・期待値変更は禁止）。
- 既存モジュールは import・変更しない。本スクリプトは自己完結のまま保つこと。
